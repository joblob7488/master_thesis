This chapter presents the measured performance obtained from the baseline and CFU-accelerated TM inference implementations on the NEORV32 RISC-V processor. The central question is whether the clause evaluation speedup reported for the NoisyXOR problem in the preceding project thesis \textcolor{red}{cite deg sjæl} carries over to MNIST, where the model is too large for on-chip memory and must reside in off-chip DDR2 SDRAM. Accuracy is reported first to confirm that correctness is preserved, after which the measurements isolate the effect of model placement and of the custom instructions on inference time.

\section{Experimental Configuration}
\label{sec:results_experimental}

All measurements were obtained on a Nexys A7 board with a Xilinx Artix-7 100T FPGA, running the NEORV32 processor at 100 MHz. The trained model performs MNIST digit classification over ten classes with 2000 clauses per class, operating on 784 binary features per image. The model is stored in the zero-padded layout described in Section~\ref{sec:literal_layout}, totaling approximately 3.81 MB, and is held in DDR2 SDRAM during inference after being transferred from Flash at boot, as described in Section \ref{sec:flash_to_ddr2}.

Three inference implementations were evaluated:

\begin{itemize}
    \item Baseline software, performing clause evaluation with standard RISC-V
    instructions.
    \item CFU Design 1, offloading the evaluation of a single chunk to a
    combinational custom instruction.
    \item CFU Design 2, offloading the full class scoring pipeline to a multiple stateful custom instructions.
\end{itemize}

Each implementation was evaluated under both load strategies described in Section~\ref{sec:inference_pipeline}: the per-chunk strategy, which reads model words directly from DDR2 using the default 8 KB DMEM, and the per-class strategy, which preloads one class into a 512 KB DMEM buffer before evaluation. This gives a total of six configurations to be evaluated.

All configurations performed classification on the same set of images, containing 20 examples of each digit, totaling 200 images. Reusing the same samples across every run keeps the comparison controlled and the measurements reproducible.

Performance measurements were taken by using the processor's cycle counter, provided by the Zicntr ISA extension, accessed via the \texttt{neorv32\_cpu\_get\_cycle()} function in C. At the 100 MHz system clock, one millisecond corresponds to 100\,000 cycles. For the per-class configurations, the counter is performs timing measurements around the full prediction, and separate measurementes around the class preload and the clause scoring, so that the total inference time can be separated into its preload and scoring components. The per-chunk configurations perform both model reads and clause evaluation within the scoring function, making it hard to separate the two in a meaningful way. The per-chunk configurations therefore have no separate measurements for model reads and scoring. Instead, a single timing is reported for the complete inference. For every configuration, the reported values are averaged over the 200-image test set.

Each configuration was measured in isolation.

\section{Accuracy Verification}
\label{sec:results_accuracy}

For the performance measurements compared later to be meaningful, each configuration must produce the same classification result on the test data set. This way, differences in timing reflects the implementation alone, and not a change in what is being computed. This section confirms that classification accuracy is maintained across the different configurations.

The accuracy of each configuration is shown in Table \ref{tab:results_accuracy}.

As we can see above, accuracy is maintained at approximately 97\% accross the different configurations. Because the classification output is deterministic, the accuracy should be constant, however the per-chunk baseline and per-class CFU Design 2 configurations have one less correct classification than the other configurations. This disparity is believed to be caused by an error in the communication between the NEORV32 processor and the host pc, as both of the outlier configurations seemingly misclassified the first sample, but did provide a valid class as the predicted class. The predicted class was instead two random characters, suggesting that the host pc misread part of a message transmitted by the processor and logged it as the predicted class.

Excluding this first sample, all configurations achieve identical accuracy, confirming that every implementation evaluates the same model and is therefore functionally equivalent for the timing comparisons that follow.

\textcolor{red}{sjekk at du har forklart early-exit bra nok tidligere}

The two strategies place the model data in different memories, resulting in different performance results, shown per sample in Figure \ref{fig:load_strategy_persample}. In the per-chunk strategy, each clause word is read from DDR2 at the moment it is needed during evaluation. The early exit means that most clauses are rejected after their first non-empty chunk, so only a small fraction of the model is read for any given inference. Despite this, the average inference takes 328.96 ms, as every word access pays the latency of a DDR2 read.

The per-class strategy removes the DDR2 reads from the evaluation loop by preloading an entire class into DMEM before scoring it. Once model data is on-chip, scoring is relatively fast, with clause evaluation accounts for only 80.32 ms, approximately a quarter of the per-chunk inference time. The preload however costs 780.71 ms, meaning total inference time rises to 861.05 ms. By not including DDR2 reads in the evaluation loop, the per-class strategy also loses the possibilty for early exit of clauses, meaning every clause must be read in its entirety, resulting a higher amount of total DDR2 reads compared to the per-chunk strategy.

The preload therefore accounts for approximately 91\% of per-class inference time, and the per-class total inference time exceeds the per-chunk total by a factor of roughly 2.6. However, in both strategies it is the movement of model data, rather than clause evaluation, that dominates inference time. 

In Figure \ref{fig:chunk_vs_score_variance} we see more clearly how the early-exit affects the timing of the strategies on a per sample basis.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{figures/chunk_vs_score_variance.png}
\caption{Per-sample timing variation of the baseline implementation, plotted at the same scale and centered around their respective means for direct comparison. (a) shows the per-chunk total inference times, and (b) shows the per-class scoring time, with the preload excluded.}
\label{fig:chunk_vs_score_variance}
\end{figure}

As seen above, Both the per-chunk total and per-class scoring times vary the same on the same samples, however the magnitude of the variance is substantially different, with the per-class scoring time being far more stable. This suggests that, for the per-chunk strategy, a majority of the variance in inference time across samples stems from how many DDR2 reads are required.

\label{sec:results_scoring}

This section isolates the effect of the custom instructions on clause evaluation. The scoring phase of the per-class strategy is used for the comparison, as it runs entirely from on-chip memory, and therefore exposes any difference between the implementations that DDR2 reads would otherwise hide. Table~\ref{tab:results_scoring} reports the average scoring time of the baseline software implementation, CFU Design 1, and CFU Design 2, together with the speedup relative to the baseline.

From Table \ref{tab:res_processor} we can see that the addition of custom instruction adds little logic to the processor. With a 8 KB DMEM size, CFU Design 2 uses 1942 LUTs and 1519 flip-flops against the 1885 LUTs and 1496 flip-flops of the baseline implementation, an increase of 57 LUTs and 23 flip-flops. \textcolor{red}{skriv noe tilsvarende om CFU design 1}. 

Increasing the DMEM size to 512 KB on the other hand has a larger effect on logic than the addition of custom instructions has. For the baseline implementation, increasing the DMEM size from 8 KB to 512 KB raises the amount of LUTs from 1885 to 2058, which is an increase of 173, which exceeds the amount added by the introduction of the CFU.  This is due to the fact that the 512 KB DMEM is too large to be held entirely in block RAM, so part of it is implemented as LUTs.

The DDR2 memory subsystem takes up the majority of the logic usage of the complete design, totaling 5726 LUTs, which is more than twice the amount of LUTs of the processor, no matter the configuration. The complete design, with both the processor and the memory subsystem, uses around 12\% of the LUTs and 6\% of the flips-flops available as shown in Table \ref{tab:res_total}, depending on the processor configuration. Block RAM, on the other hand, is fully consumed in the configurations needing 512KB DMEM. No DSP slices are used in any configuration, as clause evaluation relies only on bitwise and integer operations, as described in Section~\ref{sec:tm_inference}. In other words, block RAM is the limiting factor, which is why the model needs to reside off-chip in DDR2 and Flash memory.
