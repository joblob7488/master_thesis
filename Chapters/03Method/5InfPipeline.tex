\section{Inference Pipeline}
\label{sec:inference_pipeline}

This section describes how inference is performed once the model has been loaded into DDR2. Two strategies for accessing the model during clause evaluation were developed and evaluated: a per-chunk strategy that reads model data directly from DDR2 during evaluation, and a per-class strategy that preloads an entire class into DMEM before evaluation, with a simple visual comparison shown in Figure \ref{fig:load_strategy_comparison}. Both strategies are described below, along with a description of how the input is encoded and the output returned.

\subsection{Input Encoding and Preprocessing}
\label{sec:input_encoding}

The host PC sends each input sample as the booleanized MNIST image, packed into 25 little-endian 32-bit words. The 784 positive literal bits occupy bits 0 through 783, with the final word being zero-padded in its upper 16 bits. This results in 100 bytes being transmitted per image over UART.

Only the positive literals are transmitted. As described in Section~\ref{sec:tm_inference}, the negative literals do not need to be stored or transmitted explicitly, since each negative literal $\lnot x_i$ is obtained by taking the bitwise complement of the corresponding positive literal $x_i$. On the device, the received words are reassembled into the input array $Xi$, and the complement is applied implicitly during clause evaluation.

Clause evaluation follows the bitmask approach established in Section~\ref{sec:tm_inference}. For each 32-bit word $k$ of a clause, the positive mask $\text{pos}$ and negative mask $\text{neg}$ encode the included literals. A clause fails if any included positive literal is not satisfied, detected by $\text{pos} \;\&\; \lnot Xi[k] \neq 0$, or if any included negative literal is not satisfied, detected by $\text{neg} \;\&\; Xi[k] \neq 0$. A word in which both masks are zero contributes no constraint and is skipped. The all-exclude case, where no literals are included in the clause, is tracked separately so that an empty clause does not produce a vote.


\subsection{Per-Chunk Load Strategy}
\label{sec:per_chunk}

In the per-chunk strategy, model words are read directly from DDR2 on demand during clause evaluation. For each clause, the inner loop iterates over the 25 word positions, reading the positive and negative mask words for that position from DDR2 immediately before they are used. The most relevant parts of the evaluation loop are shown in Listing~\ref{lst:per_chunk}.

\begin{code}
\captionof{listing}{Per-chunk clause evaluation. Each mask word is read from DDR2 on demand inside the inner loop.}
\label{lst:per_chunk}
\begin{minted}{c}

uint32_t output = 1; // clause output variable    
uint32_t all_exclude = 1; // all_exclude case tracker
for (int k = 0; k < POS_CHUNKS; k++) {
    // Get action bits from DDR2
    uint32_t pos = get_pos_action(cls, j, k);  // DDR2 read
    uint32_t neg = get_neg_action(cls, j, k);  // DDR2 read

    if ((pos | neg) == 0) continue;
    // If not all zero, then set all_exclude to 0
    all_exclude = 0; 

    // Check if there are mismatches in 
    // the positive literals and negative literals
    if (pos & ~Xi[k]) { output = 0; break; } 
    if (neg &  Xi[k]) { output = 0; break; }
}
// AND clause output and state of all_exclude (negated)
output = output && !all_exclude; 
\end{minted}
\end{code}

The model is laid out in DDR2 as an array indexed first by class, then by clause, then by word. Each class occupies $2000 \times 50$ words, each clause occupies 50 words (25 positive followed by 25 negative), and the word address for a given class, clause, and chunk is computed directly from these indexes. The early-exit check breaks out of the inner loop as soon as a clause is fails, avoiding further DDR2 reads for that clause. In other words, one of the first 32 literals (either positive or negative) do no match, the remaining 752 literals are not checked.

This strategy keeps DMEM usage minimal, as only the input sample and a small amount of working state are held on-chip. The drawback is that every mask word must be fetched from DDR2 at the moment it is needed during evaluation, and each of these reads is relatively slow compared to on-chip memory access.


\subsection{Per-Class Load Strategy}
\label{sec:per_class}

In the per-class strategy, the entire set of model words for one class is copied from DDR2 into a buffer in DMEM before that class is evaluated. The clause evaluation loop then reads exclusively from this on-chip buffer, performing no DDR2 accesses during evaluation. A simplified preload and evaluation loop are shown in Listing~\ref{lst:per_class}.

\begin{code}
\captionof{listing}{Per-class strategy. One full class is copied into a DMEM 
buffer, after which clause evaluation reads only from on-chip memory.}
\label{lst:per_class}
\begin{minted}{c}
// Preload one class (2000 clauses x 50 words) into DMEM
for (uint32_t i = 0; i < WORDS_PER_CLASS; i++) {
    class_buf[i] = ddr2_read(base + i);
}

// Evaluate clauses from the DMEM buffer
for (int j = 0; j < CLAUSES; j++) {
    uint32_t output      = 1;
    uint32_t all_exclude = 1;

    const uint32_t *clause = &class_buf[j * WORDS_PER_CLAUSE];
    
    for (int k = 0; k < POS_CHUNKS; k++) {
        uint32_t pos = clause[k];
        uint32_t neg = clause[POS_CHUNKS + k];
    
        ... // The rest of the function is the
            // same as in the per-chunk strategy
    }
}
\end{minted}
\end{code}

One class consists of $2000 \times 50 = 100\,000$ words, or 400 KB. Storing a full class in DMEM therefore requires the DMEM to be expanded beyond the default size of 8 KB. The DMEM was expanded to 512 KB to accommodate the class buffer together with the program's other data. This expansion triggers a Vivado design rule error (REQP-1962) which hindered the design from being implemented. However by manually ignoring this design rule in the constraint file of the project the design was implemented and uploaded to the FPGA, and no functionality was seemingly affected. \textcolor{red}{SOURCE NEEDED}

The motivation for this strategy is to avoid the many small, slower DDR2 reads performed by the per-chunk strategy, hopefully bettering overall performance. Instead of reading individual mask words from DDR2 throughout evaluation, the per-class strategy moves all of the data for a class in one larger transfer, after which evaluation runs entirely from fast on-chip memory.



\subsection{Output and Response Format}
\label{sec:output_format}

After evaluating all clauses for all ten classes, the predicted class is selected by the argmax over class scores, as described in Section~\ref{sec:tm_inference}. The device returns the result in two parts. First, a single raw byte containing the predicted class is sent, offset by one to avoid transmitting a null byte for class zero. Second, a human-readable result line of the form \texttt{RESULT <cycles> <pred>} is sent, reporting the cycle count measured for the inference and the predicted class. The cycle count is obtained from the CPU cycle counter immediately before and after the prediction. The host benchmark script reads both the raw prediction byte and the result line, logging the prediction, correctness, and cycle count for each sample.
